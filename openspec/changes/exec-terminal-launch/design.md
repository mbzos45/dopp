## Context

- 既存のexecボタンはUIのみで、処理は未実装。
- コンテナ接続は "[docker/podman] exec -it <コンテナ名/ID> /bin/bash" を新規ターミナルで実行する必要がある。
- OSやウインドマネージャーの既定ターミナルを優先し、未検出時はフォールバック順を適用する。
- 起動失敗時にはユーザーに明示し、手動接続用のコマンドをクリップボードへ退避する。

## Goals / Non-Goals

**Goals:**
- execボタンから対象コンテナへ接続するターミナルを新規ウインドで起動する。
- OS別の優先順位に従ってターミナルを解決し、適切な起動方法でコマンドを実行する。
- 起動失敗時にエラーを表示し、手動接続用コマンドをクリップボードへコピーする。

**Non-Goals:**
- ターミナル内での操作支援や、コンテナ内部のUX改善。
- 端末アプリごとの高度なカスタム（テーマ/プロファイル/分割起動など）。
- execコマンド以外の接続方式（sshやattach等）の追加。

## Decisions

1. **OS別の起動戦略を明文化して実装する**
   - 理由: OSごとに標準のターミナル解決手段が異なるため。
   - 代替案: 単一のクロスプラットフォームライブラリで起動。
     - 見送り: 既定ターミナル優先の要件を満たせない可能性が高い。

2. **ターミナル解決は優先順位付きフォールバックで実施**
   - 理由: 既定が設定されていない環境でも一定の確実性を持たせるため。
   - 代替案: 最初に見つかったターミナルのみ使用。
     - 見送り: 起動失敗時の復旧が弱い。

3. **起動失敗時はクリップボードへ接続コマンドをコピーする**
   - 理由: 端末起動に失敗してもユーザーが即時に手動接続できるようにするため。
   - 代替案: 失敗時はエラートーストのみ。
     - 見送り: 回復導線が弱い。

4. **execコマンド生成を専用ユーティリティに集約する**
   - 理由: docker/podmanの切替や引数の拡張に備えた責務分離。
   - 代替案: UIイベント内で直接組み立て。
     - 見送り: テストや再利用が難しくなる。

## Risks / Trade-offs

- [OS固有の挙動差] → 主要OS別の分岐を分離し、失敗時にクリップボード退避で回復導線を確保する。
- [外部コマンド呼び出し失敗] → 標準出力/標準エラーをログに残し、ユーザーに明示する。
- [podman/dokerの判定揺れ] → 検出ルールを固定し、必要であれば設定で切替可能にする。
- [arboard依存によるOS権限問題] → 失敗時はエラー表示のみで継続できるようにする。

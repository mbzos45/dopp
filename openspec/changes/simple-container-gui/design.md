## Context

- 現状は `eframe` + `egui` のサンプル UI のみで、コンテナ操作 UI は未実装。
- 参考実装として `container_executer.py` があり、同様の操作感を持つ軽量 GUI を目指す。
  https://github.com/TeamSOBITS/docker_ws/blob/multi-stage/container_executer.py を参考にする。

- Docker へのアクセスには Rust のクレートを追加する必要がある。
- 使用クレートは該当バージョンの docs を参照し、非推奨 API を避ける。

## Goals / Non-Goals

**Goals:**
- コンテナ一覧を 1 行ずつ表示し、状態と名前を分かりやすく配置する。
- 状態に応じたボタン表示制御（停止中は restart/exec を非表示）を行う。
- close/refresh を上部に配置し、refresh でコンテナ一覧を再取得する。
- コンテナ数に合わせてウィンドウ高さを自動調整し、増加時は拡張する。
- exec は UI のみ実装し、処理は未実装のままにする。

**Non-Goals:**
- コンテナへのアタッチターミナル起動の実装（exec の実処理）。
- 高度なフィルタ、検索、ログ表示などの管理機能。
- 複数ホストやリモート Docker のサポート。

## Decisions

1. **Docker クライアントの選定**
   - 可能性: `bollard` などの Rust Docker クライアント。
   - **採用理由**: Docker API と非同期取得が可能で、一般的に利用実績がある。
   - **代替案**: CLI 呼び出しによる `docker` コマンド実行。
   - **不採用理由**: 出力解析が必要になり堅牢性が下がる。

2. **状態取得とリフレッシュ**
   - refresh ボタンで一覧を再取得し、内部の状態リストを更新する。
   - 取得エラーは UI に軽いエラー表示、もしくはログで通知する。

3. **ウィンドウサイズの自動調整**
   - コンテナ数に応じた高さを計算し、egui の viewport 設定で反映する。
   - refresh で増加した場合のみ高さを延長する。

4. **exec ボタンの未実装**
   - UI の配置とクリック検知まで実装し、処理は no-op にする。
   - 将来実装のために明示的な TODO を残す。

## Risks / Trade-offs

- Docker クライアントの API 差分や非推奨メソッドの混入
  → docs 参照で利用 API を固定し、必要ならバージョン固定。
- コンテナ数が多い場合の UI 高さ増加で画面外に出る
  → 最大高さやスクロールの導入を将来検討。
- refresh の頻度が高い場合の負荷
  → 手動更新のみとし、自動ポーリングは行わない。

## Migration Plan

- 新規機能のみ追加のため、既存挙動への影響は小さい。
- リリース後に UI の不具合があれば、ボタン機能を無効化する暫定対応が可能。

## ADDED Requirements

### Requirement: UIスレッドからの同期的な待機を排除する
UIはDocker操作を同期的に待機してはならない。UIスレッドからは非同期タスクの起動のみを行い、結果受信はメッセージ経由で行うこと。

#### Scenario: UI操作が即時に返る
- **WHEN** ユーザーが一覧更新やstart/stop/restartを実行する
- **THEN** UIスレッドはブロックせず、操作後もUIは入力に反応し続ける

### Requirement: 非同期タスクでDocker操作を実行する
Dockerの一覧取得、start/stop/restartはバックグラウンドの非同期タスクで実行されなければならない。

#### Scenario: 一覧更新が非同期で行われる
- **WHEN** 一覧更新タスクが起動される
- **THEN** Docker API呼び出しはバックグラウンドで実行され、UIスレッドは待機しない

### Requirement: UI状態はメッセージ経由で更新する
非同期タスクは結果を`UiEvent`のようなメッセージで通知し、UIスレッドでコンテナ一覧、エラー、ロード中フラグを更新しなければならない。

#### Scenario: 結果到達時にUI状態が反映される
- **WHEN** 非同期タスクが完了し結果メッセージを送信する
- **THEN** UIスレッドは受信メッセージを処理して状態を更新する

### Requirement: UIは結果到達時に再描画される
結果メッセージ到達時に`egui::Context::request_repaint()`を呼び、UIが即時に更新されなければならない。

#### Scenario: 非同期結果で即時に再描画される
- **WHEN** 非同期タスクが結果メッセージを送信する
- **THEN** UIは再描画要求を受け、表示が更新される

### Requirement: コンテナ単位のロード中フラグを管理する
UIはstart/stop/restartの各操作に対して、対象コンテナ単位のロード中フラグを保持しなければならない。

#### Scenario: 操作中のコンテナだけがロード中になる
- **WHEN** あるコンテナに対して操作が開始される
- **THEN** そのコンテナのロード中フラグのみが有効になる

### Requirement: 操作結果に関わらず一覧を更新する
start/stop/restartの完了時、成功・失敗に関わらず最新状態を反映するために一覧更新を行わなければならない。

#### Scenario: 失敗時でも一覧が更新される
- **WHEN** start/stop/restartの操作が失敗で完了する
- **THEN** UIは一覧更新タスクを起動し最新状態を取得する

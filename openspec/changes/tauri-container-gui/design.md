## Context

Tauri + Svelteでホスト環境向けのコンテナ操作GUIを提供する。Rust側はBollardでDocker Engine APIを利用し、Docker/Podmanの両方で動作することを前提とする。既存のGUIは最小限の操作に絞り、ターミナル自動起動は見送り、代替として"docker exec -it"のコマンドをクリップボードへコピーする。

## Goals / Non-Goals

**Goals:**
- コンテナ一覧と状態表示、起動/停止/再起動などの基本操作をGUIから行える。
- Docker/Podman互換のエンジン接続によりクロスプラットフォームで利用できる。
- ターミナル自動起動の代替として、対象コンテナの"docker exec -it"をクリップボードにコピーできる。
- ロギングはTauriの提供する仕組み、またはTauri依存の標準的なロギングを利用する。

**Non-Goals:**
- コンテナ内でこのアプリを利用すること。
- 高度な運用機能(ネットワーク/ボリューム管理、ログストリーミング、監視ダッシュボード)。
- 端末の自動起動やアタッチの完全自動化。

## Decisions

- **RustバックエンドにBollardを採用**: Docker Engine APIに公式互換のRustクライアントを使うことで、Docker/Podman双方への対応を保ちやすい。
  - 代替案: CLIラップ/プロセス呼び出し。APIの統一性・エラーハンドリングの一貫性を優先して不採用。
- **Tauriコマンド経由でUI操作を実装**: SvelteからTauriコマンドで一覧取得/操作実行を行い、責務を分離する。
  - 代替案: すべてフロントエンドでCLI実行。権限やセキュリティの観点で不採用。
- **ターミナル起動は保留し、クリップボードコピーを提供**: OS依存の端末起動差異を回避し、最小機能で用途を満たす。
  - 代替案: OS別の端末起動実装。初期段階の複雑性を避けるため不採用。
- **ログはTauri提供の仕組みを優先**: 依存増を抑えつつ、既存のログ基盤に合わせる。
  - 代替案: 独自のログライブラリ導入。初期は不要と判断。

## Risks / Trade-offs

- [Docker/Podman環境差] → 接続設定やソケットパス差異を吸収する設定とエラーメッセージを用意する。
- [権限・アクセスエラー] → 起動時に接続検証を行い、UIに明確なフィードバックを表示する。
- [CLI代替のUX制限] → クリップボードコピーを提供し、将来の端末起動機能へ拡張可能な設計にする。
- [依存追加によるビルド負荷] → Bollard/anyhowのみに限定し、機能拡張は段階的に行う。

## Migration Plan

- 新規追加のため移行は不要。初期リリースでは既存機能への影響がないことを確認する。
- もし接続設定が必要になった場合は、設定画面/環境変数を追加し後方互換を維持する。
